# 工具选型与冲突矩阵

> 先解决明确的问题，再增加组件。多个“记忆/自进化”工具并不天然叠加，重复写入和重复调度通常比能力不足更危险。

| 工具/方向 | 解决什么 | 不解决什么 | 数据与副作用 | 与现有系统的关系 | 推荐使用方式 |
|---|---|---|---|---|---|
| 本地记忆插件 | 结构化存储、检索、可视化 | 不自动保证记忆正确或无冲突 | 可能写入本地数据库；部分实现会调用宿主模型 | 可能与现有记忆层双写 | 先只读试用，确认唯一写入源后再启用 |
| EvoMap Evolver | 从运行信号生成演化建议和审计记录 | 不会可靠地自动修复生产系统 | Hub 模式可能联网；loop 会持续运行 | 可能和已有自学习、cron、审查机制重叠 | 优先 `--review`；不要默认运行 `--loop` |
| Self-Improving Agent | 将错误、反馈和经验沉淀为候选规则/技能 | 不替代测试、代码审查或安全审批 | 会写学习记录；可能触发后续 Skill 创建 | 可与现有复盘机制互补，也可能重复记忆 | 保留人工验收和最小变更门槛 |
| Dream / 记忆整理 | 压缩、聚合、淘汰过时信息 | 不应替代精确历史召回 | 会修改记忆存储或摘要 | 与其他记忆整理器同时运行会互相覆盖 | 只保留一个主整理器 |
| 任务/定时巡航 | 监控状态、定时检查、主动提醒 | 不保证执行结果正确 | 长期运行、可能外发消息或消耗额度 | 易与 heartbeat、cron 重复 | 一个目标只保留一个调度源，并设置降噪门槛 |

## 决策顺序

1. 写清当前问题：记不住、检索不准、重复犯错，还是缺少审计？
2. 盘点已经启用的组件，避免同一职责有两个写入/调度源。
3. 先做只读或 dry-run 验证；记录输入、输出、写入位置和网络行为。
4. 只有确认收益大于冲突成本时，才安装、改配置或启动长期循环。
5. 任何新自动化都要有停止方式、日志位置、失败通知和回滚说明。
